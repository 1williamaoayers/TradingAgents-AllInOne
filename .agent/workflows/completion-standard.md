# 完成标准检查清单

## 触发条件
当我准备说"完成"、"已完成"、打勾 `[x]` 时，必须先通过此检查清单

---

## 核心原则
**功能完全可用才算完成**

---

## 自检清单

### ❌ 这些不算完成

在说"完成"前，逐项检查：

- [ ] 代码能运行（但功能不完整）
  - 例：登录接口能调用，但没有密码验证

- [ ] 界面能显示（但交互有问题）
  - 例：按钮显示了，但点击没反应

- [ ] 没有报错（但结果不对）
  - 例：程序运行了，但数据是错的

- [ ] 我觉得应该可以（但没测试）
  - 例：代码写完了，但没实际运行过

**如果任何一项为 YES，不算完成，继续工作**

---

### ✅ 完成的标准

必须同时满足以下所有条件：

- [ ] 功能完全可用（所有需求都实现）
- [ ] 已实际测试验证（有真实输出/截图/日志）
- [ ] 用户可以立即使用（不需要额外修复）

---

## 验证要求

### 必须有证据

汇报格式：
```markdown
✅ [功能名称] 已完成并验证

**实现内容**：
- [具体实现了什么]

**验证证据**：
根据 [文件路径] L[行号]:
[引用实际代码]

测试结果：
[真实输出/截图/日志]

因此可以确认功能完全可用。
```

---

### 禁止的表述

- ❌ "应该可以"
- ❌ "待验证"
- ❌ "理论上没问题"
- ❌ "看起来正常"
- ❌ "大概率没问题"
- ❌ "基本完成"

**没有证据 = 禁止说完成**

---

## 如果未通过检查

1. 继续工作直到真正完成
2. 进行实际测试验证
3. 获取真实证据
4. 再次检查此清单
5. 通过后才能汇报

---

## 示例对比

### ❌ 错误示范
```
我：登录功能已完成！
用户：能用吗？
我：应该可以，我写完代码了
用户：你测试了吗？
我：还没，但代码没报错
```
❌ 没有验证就说完成

### ✅ 正确示范
```
我：✅ 登录功能已完成并验证

**实现内容**：
- 用户名密码验证
- JWT token 生成
- 登录状态保持

**验证证据**：
根据 src/api/auth.ts L25-40:
[引用实际代码]

测试结果：
```
POST /api/login
Request: {"username": "test", "password": "123456"}
Response: {"token": "eyJhbGc...", "user": {...}}
Status: 200 OK
```

已实际测试，功能完全可用。
```
✅ 有证据，可以确认完成
